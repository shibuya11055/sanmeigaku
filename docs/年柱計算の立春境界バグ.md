# 年柱計算の立春境界バグ

> 発見日: 2026-05-14
> 発見経緯: 「天中殺の波動」機能の追加レビュー中、独自実装した `eto_year` と既存 `LunarCalendarEntry.current_year?` の挙動を実測比較した際に判明
> ステータス: **未対応**（既存ユーザーの命式データに影響するため、修正前に対応方針の決定が必要）

---

## 要約（TL;DR）

`LunarCalendarEntry.current_year?` の最終行が `<=` になっているが、コメント通り `>=` であるべき。結果として、**2月1日〜2月6日生まれ**の人の **年柱（年番号 / `sexagenary_cycle_year_id`）が誤って算出される**。

立春前後の約5〜6日間（年によって変動）の誕生日で発生。誕生日が立春前なら本来「前年の干支」だが「今年の干支」が割り当てられる。立春後なら逆。

影響範囲は年柱経由で連鎖し、**陰占の年柱・年運・位相法・干合・異常干支** すべてに波及する（天中殺タイプ自体は日柱から決まるため影響なし）。

---

## 該当箇所

### バグ本体

[app/models/lunar_calendar_entry.rb:185-197](../app/models/lunar_calendar_entry.rb#L185)

```ruby
def self.current_year?(birth_date)
  year = birth_date.year
  boundary_date = Date.new(year, 2, 7)

  # 2月7日以降であれば干支歴で今年であることが確定する（2044年までのデータで）
  return true if birth_date >= boundary_date

  # 1月であれば干支歴で去年であることが確定する
  return false if birth_date.month == 1

  # 誕生日が2月の初日以降であれば今年である
  birth_date.day <= self.find_by(year: year).entries[1]   # ← ここの `<=` が `>=` であるべき
end
```

最後の行のコメントは正しい：「**2月の初日（＝立春日）以降であれば今年である**」。だが実装は `<=`（立春日以前なら今年）になっており、コメントと逆。

### 呼び出し元

[app/services/fortune_analysis_calculator.rb:31](../app/services/fortune_analysis_calculator.rb#L31)

```ruby
def build_year_num
  year_num = (year - 1900 + 37) % 60
  return 60 if year_num.zero?
  # 生まれ年が去年の場合は1年引く
  year_num -= 1 if !LunarCalendarEntry.current_year?(birth_date)
  year_num = 60 if year_num.zero?
  year_num
end
```

`current_year?` が false（＝立春前 / 前年の干支歴）と判定された場合に `year_num` を 1 引いて前年扱いにしている。`current_year?` がバグっているため、立春前後の判定が逆転している。

呼び出し元はこの1箇所のみ（`grep -rn "current_year?" app/ lib/` で確認）。

## 症状（実測）

2025年（立春 = 2月3日、`entries[1]=3`）を例にとると：

| 誕生日 | 立春との位置 | 本来の干支歴年 | `current_year?` が返す値 | `year_num` の補正 | バグ後の干支歴年 | 結果 |
|---|---|---|---|---|---|---|
| 2025/2/1 | 立春前 | 2024 | **true** | しない | 2025 | ❌ 誤 |
| 2025/2/2 | 立春前 | 2024 | **true** | しない | 2025 | ❌ 誤 |
| 2025/2/3 | 立春当日 | 2025 | true | しない | 2025 | ✅ 正 |
| 2025/2/4 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/5 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/6 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/7 | Feb 7+ 短絡 | 2025 | true | しない | 2025 | ✅ 正 |
| 2025/1/30 | 1月 | 2024 | false | する | 2024 | ✅ 正 |
| 2025/3/1 | 3月以降 | 2025 | true | しない | 2025 | ✅ 正 |

2025/2/3（立春当日）は **偶然** `3 <= 3` が成立して正しい値を返す。それ以外の Feb 1-2, 4-6 がすべて誤判定。

### 具体的な命式の崩れ例

**2025/2/4 生まれ**の場合：
- 干支歴上は 2025年 2月（立春後）→ 乙巳年 戊寅月
- 現在の実装では year_num が 1 引かれる → 2024年扱い → 甲辰年として算出される
- **年柱の干支がまるごと１個前にズレる**

---

## 影響範囲

`sexagenary_cycle_year_id`（年柱）が誤ると、以下が連鎖的に誤る：

| 機能 | 影響 |
|---|---|
| **陰占図の年柱表示** | 年干・年支がそもそも違う干支に |
| **年柱の蔵干** | 蔵干計算の入力が違う支になる |
| **異常干支判定** | 年柱が異常干支かどうかの判定が変わる |
| **干合（年×日 / 年×月）** | 年干が違うので干合の有無が変わる |
| **位相法** | 年支が違うので三合・支合・刑・冲などすべて変わる |
| **年運の相性計算** | 年柱から年干/年支への相性が違ってくる |
| **数理法** | 年柱の蔵干を含むので五行集計値が変動 |
| **大運の各旬の関係性** | 大運 vs 年柱の対冲・支合判定が変わる |

**影響を受けない機能**：
- **天中殺タイプ**: 日柱から決まるため不変
- **日柱・月柱**: それぞれの計算ロジックは別（`build_day_num` / `build_month_num` は無関係）
- **「天中殺の波動」**（本機能）: 日柱依存なので不変
- **陽占図**: 日干と各位の干（蔵干）で決まる。年柱の蔵干が変わると一部の主星も変わる可能性あり

---

## 発生条件

`LunarCalendarEntry` の 1925-2044 のデータを見ると、`entries[1]`（2月の月入日 = 立春日）はほぼ全年で **3〜6** の値を取る。よって：

- **誤判定が起こる日数**: 各年で約 **5日**（立春日のみ偶然正解。前後の Feb 1-2 と Feb 4-6 が誤判定）
- **データ範囲全体（120年）の影響日数**: 約 **600日分の誕生日**

例外: `entries[1]` が **3** の年（2021, 2025, 2029...）は立春前の誤判定日が Feb 1-2 の2日、立春後の誤判定日が Feb 4-6 の3日、計5日。`entries[1]` が **5** の年（少数）は Feb 1-4 と Feb 6 の5日。

## 修正案

### コード修正

[app/models/lunar_calendar_entry.rb:196](../app/models/lunar_calendar_entry.rb#L196)

```diff
- birth_date.day <= self.find_by(year: year).entries[1]
+ birth_date.day >= self.find_by(year: year).entries[1]
```

これでコメント「2月の初日以降であれば今年である」と挙動が一致する。

### 修正後の挙動検証

修正後、上記再現スクリプトの全行が ✓ になる。

なお、本リポジトリには既に正しい実装の参考として [app/services/heavenly_void_wave_calculator.rb](../app/services/heavenly_void_wave_calculator.rb) の `eto_year` 関数がある（private メソッド）。挙動はこれと同じになる。中長期的にはこちらを `LunarCalendarEntry` のクラスメソッドに昇格して共通化するのが理想。

## 修正時の注意：DB マイグレーションが必要

### 影響を受ける既存レコード

`fortune_analysis` テーブルに既に保存されている命式のうち、`birthday` が **2月1日〜2月6日** のレコードは、`sexagenary_cycle_year_id` が誤った値で永続化されている可能性が高い。

```sql
SELECT id, birthday, sexagenary_cycle_year_id
FROM fortune_analyses
WHERE EXTRACT(MONTH FROM birthday) = 2
  AND EXTRACT(DAY FROM birthday) BETWEEN 1 AND 6;
```

### 推奨マイグレーション方針

**Option A: 該当レコードを削除して再生成**（シンプル）
1. 上記 SQL で該当 `fortune_analysis` の `id` を抽出
2. 関連する `fortune_records`（鑑定カルテ）の `fortune_analysis_id` 紐付けをチェック（FK 参照あれば一旦外す）
3. 該当 `fortune_analysis` を `destroy`
4. 次回ユーザーがその誕生日でアクセスしたら自動で再計算・再生成される

**Option B: rake task で in-place 修正**（影響範囲を最小化）
1. 該当レコードに対し、`FortuneAnalysisCalculator.call(birthday)` で正しい値を再算出
2. `sexagenary_cycle_year_id` のみを `update_columns` で書き換え（callbacks をスキップ）

```ruby
# lib/tasks/fix_year_pillar_boundary.rake
namespace :fortune_analysis do
  task fix_year_pillar_boundary: :environment do
    affected = FortuneAnalysis.where(
      "EXTRACT(MONTH FROM birthday) = 2 AND EXTRACT(DAY FROM birthday) BETWEEN 1 AND 6"
    )
    puts "対象: #{affected.count} 件"
    affected.each do |fa|
      correct = FortuneAnalysisCalculator.allocate.tap { |c| c.instance_variable_set(:@birth_date, fa.birthday) }.send(:build_year_num)
      next if correct == fa.sexagenary_cycle_year_id
      puts "id=#{fa.id} birthday=#{fa.birthday}: #{fa.sexagenary_cycle_year_id} → #{correct}"
      fa.update_columns(sexagenary_cycle_year_id: correct)
    end
  end
end
```

※ 上記は雛形。`FortuneAnalysisCalculator` の private メソッド呼び出しは作り直すか、専用の公開 API を作るほうが綺麗。

### `fortune_records`（鑑定カルテ）への波及

`fortune_record` テーブルが `fortune_analysis_id` を参照している場合、Option A で `fortune_analysis` を削除すると外部キー違反になる可能性がある。実装を確認のうえ、どちらの方針が安全か判断する。

## 対応の優先度判断材料

- **占術精度の観点**: 致命的。年柱が違えば命式全体が違う鑑定になる
- **影響を受ける誕生日範囲**: 1925-2044 のうち各年で Feb 1-6 の約5日 → 全体で約600日分
- **修正コストの観点**: コード修正は1文字（`<=` → `>=`）。だが既存DBレコードの再生成が必要
- **後方互換性の懸念**: 既存ユーザーが「自分の命式が突然変わった」と感じる可能性。修正前にユーザー通知や、自分（プロジェクトオーナー本人）の鑑定実績への影響を確認すべき

修正タイミングは、現状のクライアント・鑑定カルテ件数や、自分の運用上の都合を見て決める。
