# 年柱計算の立春境界バグ

> 発見日: 2026-05-14
> 発見経緯: 「天中殺の波動」機能の追加レビュー中、独自実装した `eto_year` と既存 `LunarCalendarEntry.current_year?` の挙動を実測比較した際に判明
> 対応日: 2026-07-18
> ステータス: **コード修正・回帰テスト追加済み**（旧DB計算テーブルの削除マイグレーションを追加済み）

---

## 要約（TL;DR）

`LunarCalendarEntry.current_year?` の2月判定が `<=` になっていたが、コメント通り `>=` であるべき。結果として、**2月1日〜2月6日生まれ**の人の **年柱（年番号 / `sexagenary_cycle_year_id`）が誤って算出される**。

立春前後の約5〜6日間（年によって変動）の誕生日で発生。誕生日が立春前なら本来「前年の干支」だが「今年の干支」が割り当てられる。立春後なら逆。

影響範囲は年柱経由で連鎖し、**陰占の年柱・年運・位相法・干合・異常干支** すべてに波及する（天中殺タイプ自体は日柱から決まるため影響なし）。

---

## 該当箇所

### バグ本体

[app/models/lunar_calendar_entry.rb](../app/models/lunar_calendar_entry.rb)

```ruby
def self.current_year?(birth_date)
  year = birth_date.year
  return false if birth_date.month == 1
  return true if birth_date.month > 2

  # 2月は月入日当日以降を干支歴の当年として扱う
  birth_date.day >= find_by(year: year).entries[1]
end
```
2月の月入日当日以降を当年、それより前を前年として扱う。

### 呼び出し元

[app/models/sanmeigaku/static_data.rb](../app/models/sanmeigaku/static_data.rb) と
[app/services/sanmeigaku/natal_chart_calculator.rb](../app/services/sanmeigaku/natal_chart_calculator.rb)

```ruby
def build_year_num
  year_num = (birth_date.year - 1900 + 37) % 60
  return 60 if year_num.zero?
  # 生まれ年が去年の場合は1年引く
  year_num -= 1 if !LunarCalendarEntry.current_year?(birth_date)
  year_num = 60 if year_num.zero?
  year_num
end
```

`current_year?` が false（＝立春前 / 前年の干支歴）と判定された場合に `year_num` を 1 引いて前年扱いにしている。`current_year?` がバグっているため、立春前後の判定が逆転している。

現在の呼び出し元は `Sanmeigaku::NatalChartCalculator#build_year_num` である。

## 症状（実測）

2025年（立春 = 2月3日、`entries[1]=3`）を例にとると：

| 誕生日 | 立春との位置 | 本来の干支歴年 | `current_year?` が返す値 | `year_num` の補正 | バグ後の干支歴年 | 結果 |
|---|---|---|---|---|---|---|
| 2025/2/1 | 立春前 | 2024 | **true** | しない | 2025 | ❌ 誤 |
| 2025/2/2 | 立春前 | 2024 | **true** | しない | 2025 | ❌ 誤 |
| 2025/2/3 | 立春当日 | 2025 | true | しない | 2025 | ✅ 正 |
| 2025/2/4 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/5 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/6 | 立春後 | 2025 | **false** | する | 2024 | ❌ 誤 |
| 2025/2/7 | Feb 7+ 短絡 | 2025 | true | しない | 2025 | ✅ 正 |
| 2025/1/30 | 1月 | 2024 | false | する | 2024 | ✅ 正 |
| 2025/3/1 | 3月以降 | 2025 | true | しない | 2025 | ✅ 正 |

2025/2/3（立春当日）は **偶然** `3 <= 3` が成立して正しい値を返す。それ以外の Feb 1-2, 4-6 がすべて誤判定。

### 具体的な命式の崩れ例

**2025/2/4 生まれ**の場合：
- 干支歴上は 2025年 2月（立春後）→ 乙巳年 戊寅月
- 現在の実装では year_num が 1 引かれる → 2024年扱い → 甲辰年として算出される
- **年柱の干支がまるごと１個前にズレる**

## 影響範囲

`sexagenary_cycle_year_id`（年柱）が誤ると、以下が連鎖的に誤る：

| 機能 | 影響 |
|---|---|
| **陰占図の年柱表示** | 年干・年支がそもそも違う干支に |
| **年柱の蔵干** | 蔵干計算の入力が違う支になる |
| **異常干支判定** | 年柱が異常干支かどうかの判定が変わる |
| **干合（年×日 / 年×月）** | 年干が違うので干合の有無が変わる |
| **位相法** | 年支が違うので三合・支合・刑・冲などすべて変わる |
| **年運の相性計算** | 年柱から年干/年支への相性が違ってくる |
| **数理法** | 年柱の蔵干を含むので五行集計値が変動 |
| **大運の各旬の関係性** | 大運 vs 年柱の対冲・支合判定が変わる |

**影響を受けない機能**：
- **天中殺タイプ**: 日柱から決まるため不変
- **日柱・月柱**: それぞれの計算ロジックは別（`build_day_num` / `build_month_num` は無関係）
- **「天中殺の波動」**（本機能）: 日柱依存なので不変
- **陽占図**: 日干と各位の干（蔵干）で決まる。年柱の蔵干が変わると一部の主星も変わる可能性あり

---

## 発生条件

`LunarCalendarEntry` の 1925-2044 のデータを見ると、`entries[1]`（2月の月入日 = 立春日）はほぼ全年で **3〜6** の値を取る。よって：

- **誤判定が起こる日数**: 各年で約 **5日**（立春日のみ偶然正解。前後の Feb 1-2 と Feb 4-6 が誤判定）
- **データ範囲全体（120年）の影響日数**: 約 **600日分の誕生日**

例外: `entries[1]` が **3** の年（2021, 2025, 2029...）は立春前の誤判定日が Feb 1-2 の2日、立春後の誤判定日が Feb 4-6 の3日、計5日。`entries[1]` が **5** の年（少数）は Feb 1-4 と Feb 6 の5日。

## 再現手順

```ruby
# bin/rails runner で実行
dates = (1..7).map { |d| Date.new(2025, 2, d) }
dates.each do |d|
  cy = LunarCalendarEntry.current_year?(d)
  feb_boundary = LunarCalendarEntry.find_by(year: 2025).entries[1]
  should_be_current = d.day >= feb_boundary
  status = cy == should_be_current ? '✓' : '✗'
  puts "#{d}: current_year?=#{cy} (期待: #{should_be_current}) #{status}"
end
```

## 修正案

### コード修正

[app/models/lunar_calendar_entry.rb:196](../app/models/lunar_calendar_entry.rb#L196)

```diff
- birth_date.day <= self.find_by(year: year).entries[1]
+ birth_date.day >= self.find_by(year: year).entries[1]
```

これでコメント「2月の初日以降であれば今年である」と挙動が一致する。

### 修正後の挙動検証

修正後、上記再現スクリプトの全行が ✓ になる。

なお、本リポジトリには既に正しい実装の参考として [app/services/heavenly_void_wave_calculator.rb](../app/services/heavenly_void_wave_calculator.rb) の `eto_year` 関数がある（private メソッド）。挙動はこれと同じになる。中長期的にはこちらを `LunarCalendarEntry` のクラスメソッドに昇格して共通化するのが理想。

## 既存DBレコードについて

命式計算を静的マスタへ移行したため、`fortune_analyses` の事前生成レコードと補正処理は廃止した。
既存DBを更新する場合は、先に必要な鑑定履歴をバックアップしたうえで、
`RemoveLegacyCalculationTables` マイグレーションを適用して旧計算テーブルを削除する。
以後の命式は `Sanmeigaku::NatalChartCalculator.call(birth_date: date)` で都度算出する。

## テスト方針

`LunarCalendarEntry.current_year?` の2月境界と、`Sanmeigaku::NatalChartCalculator` の年柱番号を
モデル・サービステストで検証する。DBの事前生成データを前提にした統合テストは行わない。

## 対応の優先度判断材料

- **占術精度の観点**: 致命的。年柱が違えば命式全体が違う鑑定になる
- **影響を受ける誕生日範囲**: 1925-2044 のうち各年で Feb 1-6 の約5日 → 全体で約600日分
- **修正コストの観点**: コード修正は1文字（`<=` → `>=`）。だが既存DBレコードの再生成が必要
- **後方互換性の懸念**: 既存ユーザーが「自分の命式が突然変わった」と感じる可能性。修正前にユーザー通知や、自分（プロジェクトオーナー本人）の鑑定実績への影響を確認すべき

修正タイミングは、現状のクライアント・鑑定カルテ件数や、自分の運用上の都合を見て決める。

# 既存 current_year? と、今回作った eto_year（heavenly_void_wave_calculator の private）の挙動比較
